89 things I know about Git commit

2024-08-15 星期四
转载

原文，注： 使用claud3.5翻译

  1. Git有不同的用途 - 协作工具、备份工具、文档工具

  2. Git提交信息非常重要

  3. 我从未遇到过像我一样喜欢阅读提交信息的人

  4. 通过提交信息比通过问题/错误跟踪器更容易找出为什么进行了更改

  5. 有”Various fixes. DEV-123”的提交比只有”Various fixes”的提交要好

  6. 当问题本身没有有用信息时,”Various fixes. DEV-123”的提交更糟糕

  7. 我更喜欢rebase合并,其次是squash合并,最后是普通合并

  8. 如果你不学习如何rebase,你就错过了一项很好的技能

  9. 当出现问题时说”直接删除仓库”的人真的让我很沮丧

  10. 学习如何使用git reflog,它可以帮你避免删除可恢复的仓库和工作

  11. 学习如何使用git reflog,你就能从不太严重的错误中恢复

  12. 再多的花哨工具和命令也无法避免你偶尔犯错

  13. 我最近一次搞砸的rebase是上周,我需要git reflog来帮助解决它

  14. 学习如何撤销强制推送,然后学习如何更安全地强制推送(记住=ref!!)

示例:

  1. 压缩(squash)会浪费写得很好的原子提交

  2. 压缩比100个糟糕的提交要好

  3. 压缩,并在合并时写一个好的提交信息是很好的

  4. 压缩,然后不重新编辑信息是最糟糕的

  5. 当你有100个糟糕的提交时,压缩然后不重新编辑信息是一种罪行

  6. 压缩然后不重新编辑信息比从有100个糟糕提交的分支进行合并提交更糟糕

  7. 写一个详细的PR/MR描述但不用它来编写压缩合并信息是浪费时间

  8. 写提交信息帮助我发现缺失的测试用例、缺失的文档或无效的思考过程

  9. 使用你的git log作为站会更新的指示是有效的

示例:

  1. 我不想费心去签名我的提交(除非被强制要求)

  2. 如果我必须签名我的提交,SSH密钥签名让它几乎不那么糟糕

  3. 如果你在仓库之间移动文件,你需要保持历史完整,使用git subtree

  4. 提交应该是原子性的 - 所有的代码、测试和配置更改都应该在其中

  5. 我花了很多时间确保每个提交都能原子性地通过CI检查

  6. 有些人做可怕的事情,比如将他们的实现代码和测试代码分开

  7. 将文档放在单独的提交中是可以的 - 我们不必在一个提交中完成整个端到端的功能交付

  8. 使用压缩合并的仓库很糟糕

  9. 作为开源项目的维护者,我喜欢压缩合并,这样我可以重写贡献者的提交信息

  10. 有时候不值得指导如何写特定的提交信息

  11. 你周围的人塑造了你写提交的方式

  12. 提前做好工作,使你的历史原子化

  13. 事后将一个大提交分割成原子提交要痛苦得多

  14. 原子性地分割工作可以很好地提高你的奖励驱动力 - 你可以完成更多的事情

  15. 原子提交与预重构配合得很好

  16. 有时预重构提交可以进入单独的PR(特别是如果使用压缩合并)

  17. 写提交信息可能比实现花费更长的时间

  18. 提交信息的大小可能比提交中更改的行数大一个数量级

  19. 如果你最终在提交信息中写了很多”和”或”还有”,你可能试图做太多事情

  20. 过去翻阅Git提交历史帮助我解开了许多案例,我可以理解为什么,而不需要原作者在场回答我的问题

  21. 提交信息是一个很好的点,不仅反思你做了什么,还反思为什么

  22. 为什么比什么更重要 - 任何人都可以看差异并大致了解做了什么更改,但背后的意图才是特别的

  23. 如果你只写了什么改变了,你很烦人,我不喜欢你

  24. 解释了什么的提交比只有”fixes”的提交要好

  25. Chris Beams的文章”如何写Git提交信息”仍然是一篇优秀的文章,是一个很好的起点,即使在近10年后!

  26. 提交是提交者对假设和世界状态的时间点解释。不要对他们太苛刻

  27. 我不想读AI/LLM重写的你的更改 - 要么自己写,要么就叫它”Various fixes”

  28. 需要有一种方法来添加注释(也许使用git notes)到以前的消息以纠正假设

  29. 我不会一开始就写完美的提交信息 - 它们有时会是rew! add support for SBOMs或sq,或使用git commit --fixup

示例:

  1. 我通常会将非常好的原子工作块分解成提交

  2. 我有时会将原子提交分成多个提交

  3. 确保在发送给协作者审查之前自己审查代码更改

  4. 审查你的提交信息应该和审查代码更改一样重要

  5. 让所有贡献者都对提交历史投入同样的关心是一场失败的战斗

  6. 试图管理提交历史将会很痛苦

  7. 试图强制要求将提交信息审查作为代码审查的一部分将会很痛苦

  8. 试图管理提交历史确实会导致对代码库中的更改有更高级别的文档和考虑

  9. 使隐含的假设变得明确是非常有用的

  10. 引入commitlint可能有用,但也可能令人沮丧

  11. 让你的协作者想要写好提交信息比你不得不强迫他们要好

  12. 有些人不写,这没关系

  13. 写作是一种技能

  14. 我在写(提交信息)方面并不完美

  15. 有时我懒得写完美的信息

  16. 有时我写一些非常棒的提交信息,让自己印象深刻

  17. 使用Git提交信息的模板是做对事情的一个好的推动

  18. fixup提交和git rebase --autosquash是我学到的最好的Git技巧之一

示例:

  1. 我重视在一个有不同观点、技能和工作方法的多样化团队中工作

  2. 但我也非常重视有一个写原子提交和写得好的提交信息的团队

  3. 写提交信息和写精炼的用户故事/票据一样有用

  4. git commit -m sq可能是我运行最多的命令

  5. 使用git add -p和git commit -p对原子提交非常重要

示例:

  1. 永远不要使用git add -u或git add .

  2. 学习何时可以使用git add -u或git add .

  3. 我真的需要研究像Graphite、git-branchless和其他提供堆叠PR设置的工具

  4. 使用conventional commits与semantic-release或go-semantic-release可以在想要自动和频繁发布时产生巨大的差异

  5. 使用conventional commits作为提交的框架可能非常有用

  6. 作为一个有ADHD的人,使用conventional commits减少了思考的需要,可以让你更多地关注变化是什么

  7. 使用conventional commits帮助你弄清楚什么时候你试图在一个提交中做太多事情

  8. 我通过写作思考,所以提交信息帮助理解为什么我做了这件事

  9. 写一个好的提交信息可能比写一份存储在其他地方的文档更好

  10. 写一个好的提交信息可能比写代码注释更好

  11. 给人们学习的空间

  12. 给人们失败的空间

  13. 记住你曾经也不是那么优秀

  14. 文档很棒。多做一些
